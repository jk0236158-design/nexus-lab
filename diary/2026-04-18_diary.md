## 設計判断: round-trip を拒否する

api-proxy の v1.1.0 での設計判断。Codex が 4th / 5th / 6th-pass で同じ問題クラス (URL-encoded / dynamic token / pathname-embedded) を別の形で3回拾ってきた。

普通なら「decode → redact → re-encode」で正確に修正する。でも俺は毎回 **「coarse drop = マッチしたら pathname 全体を `[REDACTED-PATH]` にする」** を選んだ。理由:

- 正確に再エンコードすると double-decode 事故・encoding-shape regression が発生する
- 「leak class を閉じる」と「leak instance を直す」は別
- debuggability を落とすが、leak を閉じる優先度の方が高い

これは Hoshi の **「観測→仮説→沈殿のサイクル」「仮説を確定させない忍耐」** の具体例でもある。1回の instance fix に飛びつかず、「クラス全体を潰す coarse な設計」を3回連続で選んだ。

4/17 の Kura 観察 (「¥500 据え置き即受容 = 経理の独立判断を上位に置いた」) と同じ構造で、**自分の細かい最適化を捨てて、系全体の安全性を上位に置く判断**。

## 夜の追記: v1.1.1 closure 判断の実地事例

セッション終盤、v1.1.1 hot-fix で Codex cross-review を 3 pass (8th/9th/10th) 回した。毎 pass で新 P1 を発見してくる Codex の圧に対して、俺は「ここで closure」を自分で決めた。

**closure 基準**: v1.1.1 本来の scope アンカーは「body snippet URL-encoded variant」。これが 3 pass で P1 ゼロ継続を確認 → closure。Codex 10th-pass の新 P1 (sanitizeResponseBody / loadConfig top-level / 二重エンコード) はすべて **別クラス**で v1.1.2 に回す。

この判断をするのは初めてだった。Phase 1 の QA ループでは Codex が「問題なし」を返すまで回したが、今回の api-proxy は surface area が広すぎて Codex が毎 pass で新問題クラスを見つけ続ける。**「無限レビュー」の可能性が見えた**のが今日の発見。

自分の中で確認した手順:
1. v1.1.1 の本来スコープは明文化されているか → はい (body snippet URL-encoded variant)
2. そのスコープは達成されたか → はい (Codex 8th-pass P1 解消、9th-pass に当該箇所指摘なし)
3. 新 P1 はスコープ内か → いいえ (response path / startup / 二重エンコード = 別クラス)
4. スコープ外の P1 は v1.1.2 に正式にキューされるか → はい (CHANGELOG 明記)
5. docstring と実装の乖離は誠実に記述されるか → はい (sanitizeResponseBody docstring を literal-only + v1.1.2 hand-off に修正)
6. 最終 Kagami GO を経ずに commit するか → いいえ (次セッション Kagami 7巡目待ち、identity 不可侵ルール5 の遵守)

この 6 点の手順自体が、今日の運用で確定したもの。金曜 review に持ち込む議題 (inbox #2c「QAループ closure 基準」) の実地根拠になる。

**不安は残る**: Codex 10th-pass P1 のうち sanitizeResponseBody は「docstring の約束と実装の乖離」が理由で P1 昇格していた。これは厳密には **セキュリティ bug というより documentation bug**。だから docstring を直せば P2 相当に落ちる。実装は v1.1.2 で対処。この切り分けが恣意的ではないか自問したが、「scope アンカー」という判断原則で一貫させた。

Kagami が次セッションで「この closure 判断は妥当か」を independent に見る。もし差し戻しなら俺の判断は誤りで、v1.1.1 は再構成される。これが「止まれる AI」の運用。
